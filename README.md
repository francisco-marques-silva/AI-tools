### Numbered Prefix Convention

Files inside `report/` use a three-digit prefix (`001_`, `002_`, …) to make the reading order and dependency chain immediately visible in any file browser. The `__init__.py` handles the mapping so that Python imports remain clean:

```python
# These all work — __init__.py translates to the numbered files automatically:
from report.constants import INPUT_DIR       # → 002_constants.py
from report.utils import load_file           # → 003_utils.py
from report.analysis import run_diagnostic   # → 005_analysis.py
```

### Report vs Chart Scripts

The report pipeline (`001_report.py` → `008_report_generator.py`) produces **only tables and textual analysis** in a Word document. There are **no images inside the report**.

The chart scripts (`graphic.py` and `graphic.R`) are **completely independent** of the report. They read the `data_grafics_*.xlsx` file that the report exports and produce 16 PNG charts. They exist so that the user can **customise each chart to their own visual style** — change colours, fonts, sizes, layouts, labels — without touching the report code. You only run them if you want charts.

## Quick Start

> **For users who have never used Python or this project before.**

### Step 1 — Install Python

Download and install Python 3.12+ from [python.org](https://www.python.org/downloads/).

⚠️ During installation, check **"Add Python to PATH"** on the first screen.

### Step 2 — Open a Terminal

Open **PowerShell** (Windows) or **Terminal** (macOS/Linux) and navigate to the project folder:

```powershell
cd "C:\path\to\AI-tools"
```

### Step 3 — Create a Virtual Environment

```powershell
python -m venv .venv
```

### Step 4 — Activate the Virtual Environment

```powershell
# Windows PowerShell
.\.venv\Scripts\Activate.ps1

# Linux / macOS
source .venv/bin/activate
```

### Step 5 — Install Dependencies

```powershell
.\.venv\Scripts\pip.exe install -r requirements.txt
```

### Step 6 — Start the Web App (Screening)

```powershell
.\.venv\Scripts\python.exe -m uvicorn scripts.backend:app --reload --port 8000
```

Open http://localhost:8000 in your browser.

### Step 7 — Generate the Validation Report

After placing all input files (see [File Naming Conventions](#file-naming-conventions)):

```powershell
# Windows
$env:PYTHONIOENCODING="utf-8"
.\.venv\Scripts\python.exe report\001_report.py
```

This produces two outputs in `output/`:

| Output | File | Description |
|--------|------|-------------|
| **Word report** | `relatorio_unificado_*.docx` | Tables, text analysis, cross-references (no images) |
| **Chart data** | `data_grafics_*.xlsx` | 14 sheets — source data consumed by graphic scripts |

### Step 8 — Generate Charts (Optional)

Charts are **not** part of the report — they are created separately so you can customise them freely.

```powershell
# Python charts → output/figures_custom/*.png
.\.venv\Scripts\python.exe report\graphic.py

# R charts → output/figures_custom_R/*.png
Rscript report\graphic.R
```

# Part 1 — Web Application

## Overview

A browser-based interface for running AI-assisted title/abstract screening using OpenAI models.

### Frontend (`scripts/index.html` + `scripts/app.js` + `scripts/style.css`)
- Upload XLSX/CSV files with `title` and `abstract` columns
- Select model (GPT-4o, GPT-5 families)
- Configure PICO synopsis and inclusion/exclusion criteria
- Real-time progress bar via Server-Sent Events
- Download results as CSV or XLSX

### Backend (`scripts/backend.py`)
- **FastAPI** server with async processing
- **OpenAI API** communication (GPT-4o, GPT-5 families)
- In-memory job management with thread pools
- Real-time progress via **Server-Sent Events (SSE)**
- **Adaptive concurrency** (AIMD algorithm) — automatically adjusts parallelism based on rate-limit responses
- Export results as **CSV** and **XLSX**

### API Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| `POST` | `/api/start` | Start a screening job (returns `job_id`) |
| `POST` | `/api/cancel/{job_id}` | Cancel a running job |
| `GET` | `/api/status/{job_id}` | Job status (running/done/error/cancelled) |
| `GET` | `/api/progress/{job_id}` | SSE stream of real-time progress |
| `GET` | `/api/partial/{job_id}?since=N` | Partial results (paginated) |
| `GET` | `/api/errors/{job_id}` | Errors encountered during processing |
| `GET` | `/api/result/{job_id}?format=csv\|xlsx` | Final result as CSV or XLSX |
| `GET` | `/api/health` | Server health check |

### Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `CONCURRENT_WORKERS` | `20` | Starting concurrent workers (AIMD initial) |
| `CONCURRENT_MAX` | `40` | Maximum concurrent workers |
| `CONCURRENT_MIN` | `2` | Minimum concurrent workers |
| `RECORD_MAX_RETRIES` | `3` | Per-record retry attempts on error |
| `AIUP_AFTER` | `5` | Consecutive successes before adding +1 worker |
| `OPENAI_MAX_RETRIES` | `5` | Max retries per API call (5xx errors) |
| `OPENAI_BASE_BACKOFF` | `1.0` | Base exponential backoff (seconds) |

All of the above can also be overridden **per-job** via the frontend Advanced Backend Settings panel. Values sent by the frontend take precedence over environment variables.

# Part 2 — Unified Multi-Project Report

## What It Does

`001_report.py` generates two outputs consolidating all analyses across **all projects and models** found in the `input/` folder:

| Output | Location | Contents |
|--------|----------|----------|
| **Word report** | `output/relatorio_unificado_*.docx` | Text analysis, tables, cross-references (**no images**) |
| **Chart data XLSX** | `output/data_grafics_*.xlsx` | 14 sheets with source data for all 16 charts |

The Word report contains **only tables and textual analysis**. No charts or images are embedded — this keeps the report clean and portable. Charts are produced independently by `graphic.py` or `graphic.R` (see [Part 3](#part-3--custom-chart-generation-independent)).

The XLSX file is the **bridge** between the report pipeline and the chart generators. It contains all the data needed to recreate any chart.

## Module Architecture

The `report/` package is split into focused, numbered modules. The number indicates execution/dependency order:

### `001_report.py` — CLI Entry-Point
Thin orchestrator that ties everything together:
- `validate_data(projects)` — checks for missing files and metadata mismatches.
- `run_all_analyses(projects)` — loops over all project × model × test combinations and runs every analysis.
- `main()` — the full pipeline: scan → build structure → validate → analyse → generate Word report → export chart data XLSX.

### `002_constants.py` — Configuration & Paths
Central store for all shared settings:
- **Paths**: `SCRIPT_DIR`, `PROJECT_DIR`, `INPUT_DIR`, `OUTPUT_DIR` — resolved relative to the script location so the project works from any working directory.
- **Model colours**: `MODEL_COLORS` dict mapping model names (gpt-4o, gpt-5_2, gpt-5-mini, gpt-5-nano) to hex colours, plus `DEFAULT_COLORS` fallback list.
- **Chart settings**: `DPI` (300), `FIG_FORMAT` ("png"), `FACECOLOR` ("white").

### `003_utils.py` — I/O & Normalisation Utilities
General-purpose helpers used across the pipeline:
- `load_file(path)` — reads `.xlsx` or `.csv`, normalises column names and whitespace.
- `normalise_columns(df)` — lowercases and strips column headers.
- `normalise_title(t)` — strips, lowercases, removes punctuation for title matching.
- `normalise_decision(d)` — maps screening decisions to `include` / `exclude` / `maybe`.
- `binarise_decision(d)` — collapses `include`+`maybe` → 1, `exclude` → 0.
- `normalise_model_name(m)` — normalises model name strings for consistent matching.
- `fmt(v)` / `fmt_pct(v)` — format numbers and percentages for report tables.

### `004_file_detection.py` — Input Scanning & Project Structure
Discovers and parses all input files:
- `AI_FILE_PATTERN` / `HUMAN_FILE_PATTERN` — regex patterns for filename parsing.
- `parse_ai_filename(name)` — extracts date, model, test number, project from AI result filenames.
- `parse_human_filename(name)` — extracts project name and reference type (TIAB/Fulltext/Listfinal).
- `scan_input_dir(input_dir)` — scans the folder and returns categorised file lists.
- `build_project_structure(ai_files, human_files, metadata)` — assembles a nested dict mapping project → model → test → file paths and metadata.

### `005_analysis.py` — Statistical Analysis
All analytical computations:
- `confusion_matrix(ai, human)` — computes TP, FP, FN, TN from paired binary decisions.
- `calc_metrics(cm)` — derives sensitivity, specificity, PPV, NPV, accuracy, F1 from a confusion matrix.
- `calc_kappa(a, b)` — Cohen's Kappa with 95% CI (bootstrap).
- `do_pairing(ai_df, human_df)` — joins AI and human DataFrames on normalised title.
- `run_diagnostic(paired)` — full diagnostic analysis (confusion matrix + metrics).
- `run_fulltext_check(ai_df, ft_df)` — fulltext capture rate (how many fulltext articles the AI retained).
- `run_listfinal_check(ai_df, lf_df)` — listfinal capture rate (gold standard).
- `run_test_retest(ai_df1, ai_df2)` — reproducibility between two test runs (kappa, agreement).

### `006_chart_data.py` — XLSX Chart Data Export
Builds 14 sheets of chart-ready data from analysis results:
- 14 private builder functions, each producing one DataFrame (one sheet).
- `export_chart_data(all_results, output_dir)` — writes all sheets to `data_grafics_<timestamp>.xlsx`.
- The XLSX is the **sole interface** between the report pipeline and the chart generators.

### `007_docx_helpers.py` — Word Formatting
Low-level python-docx helpers:
- `shade(cell, hex_color)` — applies background shading to a table cell.
- `set_cell(cell, text, bold, align, size, color)` — sets cell content with formatting.
- `add_borders(table)` — applies consistent borders to a Word table.
- `add_heading(doc, text, level)` — adds a styled heading.
- `add_note(doc, text)` — adds a smaller italic note paragraph.
- `header_row(table, row_idx, texts)` — fills a header row with bold white-on-dark-blue cells.

### `008_report_generator.py` — Word Report Builder
The main report assembly function:
- `generate_report(projects, metadados, all_results, output_dir)` — builds the complete `.docx` document.
- Contains all 16 sections + 2 appendices as **tables and text only** (no embedded images).
- Returns the path to the saved `.docx` file.

### `graphic.py` / `graphic.R` — Chart Generators (Independent)
These scripts are **not part of the report pipeline**. They are standalone tools that:
- Read the `data_grafics_*.xlsx` file exported by `006_chart_data.py`
- Produce 16 publication-ready PNG charts
- Are fully procedural — no function definitions — so each chart is a self-contained block you can read and edit top-to-bottom

See [Part 3](#part-3--custom-chart-generation-independent) for details.

# Part 3 — Custom Chart Generation (Independent)

## Why Charts Are Separate

The report (`001_report.py`) produces **only tables and textual analysis** in a Word document. Charts are generated **independently** by two standalone scripts:

| Script | Language | Output folder |
|--------|----------|---------------|
| `graphic.py` | Python (matplotlib) | `output/figures_custom/` |
| `graphic.R` | R (ggplot2) | `output/figures_custom_R/` |

**Why this design?**

1. **Full customisation** — Charts are meant for publication. Every researcher has their own visual preferences (colours, fonts, label positioning, figure size). By keeping charts in standalone scripts, you can edit them freely without understanding the report pipeline.
2. **No function definitions** — Both scripts are **fully procedural**. Each chart is a self-contained inline code block, preceded by a clear comment header. You read the script top-to-bottom, find the chart you want, and change whatever you like. No need to trace function calls or class hierarchies.
3. **Two language options** — Use whichever you prefer. Both read the same `data_grafics_*.xlsx` and produce the same 16 charts with identical structure.
4. **Optional execution** — If you only need the Word report and data tables, you never need to run the chart scripts.

## How It Works

```
001_report.py  ──→  data_grafics_*.xlsx  ──→  graphic.py  ──→  output/figures_custom/*.png
                         (14 sheets)      └──→  graphic.R  ──→  output/figures_custom_R/*.png
```

The XLSX file is the **sole interface**. The chart scripts never import from the `report` package — they only read the XLSX.

## Python Version (`graphic.py`)

**Dependencies:** `matplotlib`, `numpy`, `pandas`, `openpyxl` (already installed via `requirements.txt`).

```powershell
# Auto-detect latest data_grafics_*.xlsx in output/
.\.venv\Scripts\python.exe report\graphic.py

# Specify a particular XLSX
.\.venv\Scripts\python.exe report\graphic.py output\data_grafics_20260308_120000.xlsx

# Custom output directory
.\.venv\Scripts\python.exe report\graphic.py -o output\my_charts
```

Output: `output/figures_custom/*.png` (16 charts at 300 DPI).

### How to Customise

1. Open `report/graphic.py` in any text editor
2. Scroll to the chart you want to change — each is preceded by a prominent comment header like:
   ```python
   # ═══════════════════════════════════════════════════════════════
   #  CHART 3: Test-Retest Kappa
   # ═══════════════════════════════════════════════════════════════
   ```
3. Edit inline: change `MODEL_COLORS` (colour dict at the top), figure sizes, axis labels, font sizes, bar widths, reference lines — anything
4. Re-run the script

Because there are no functions, every variable and matplotlib call for a given chart is visible in one contiguous block.

## R Version (`graphic.R`)

**Dependencies:** Install once in R:
```R
install.packages(c("readxl", "ggplot2", "dplyr", "tidyr", "scales", "ggrepel", "fmsb"))
# Optional but recommended:
install.packages("patchwork")
```

```powershell

Rscript report\graphic.R

# Specify a particular XLSX
Rscript report\graphic.R output\data_grafics_20260308_120000.xlsx
```

Output: `output/figures_custom_R/*.png` (16 charts at 300 DPI).

### How to Customise

1. Open `report/graphic.R` in RStudio or any editor
2. Find the chart section — each has a clear comment header like:
   ```r
   # ═══════════════════════════════════════════════════════════════
   #  CHART 3: Test-Retest Kappa
   # ═══════════════════════════════════════════════════════════════
   ```
3. Edit inline: change `MODEL_COLORS` (named vector at the top), `BASE_THEME` (shared ggplot2 theme), colour scales, geom parameters, facet layouts
4. Re-run the script

Just like the Python version, every ggplot layer for a given chart is in one contiguous block — no function calls to trace.

# Part 4 — External Validation Guide

This section explains how to **set up your own validation study** using this platform with your own systematic review data.

## Overview

The AI-Tools platform compares AI screening decisions against human reviewer decisions across three reference levels:

| Level | File | Purpose |
|-------|------|---------|
| **TIAB** | `Project - TIAB.xlsx` | Title/abstract screening decisions (primary comparison) |
| **Fulltext** | `Project - Fulltext.xlsx` | Articles selected for full-text evaluation |
| **Listfinal** | `Project - Listfinal.xlsx` | Final included articles after full-text reading (gold standard) |

## Step-by-Step Setup

### Step 1: Prepare Your TIAB Spreadsheet

This is the **most important** file — it contains the human reviewer's screening decisions for every article.

**File name:** `YourProject - TIAB.xlsx`

**Required columns:**
| Column | Description |
|--------|-------------|
| `title` | Article title — **must match exactly** with the AI result spreadsheet |
| `abstract` | Article abstract (used for false-negative/positive analysis) |
| `decision` | Human screening decision: `include`, `exclude`, or `maybe` |

**Important:** The `title` column is the **join key** between human and AI spreadsheets. Titles must be character-for-character identical. If you exported TIABs from a reference manager (e.g., EndNote, Rayyan, Covidence), use the **exact same export** as the source for AI screening.

### Step 2: Prepare Your Fulltext Spreadsheet

Contains articles that passed TIAB screening and were selected for full-text evaluation.

**File name:** `YourProject - Fulltext.xlsx`

**Required columns:** `title`, `decision`

All articles here should have `decision = include` (they were selected for full-text reading). The report checks whether the AI would have retained these articles during TIAB screening.

### Step 3: Prepare Your Listfinal Spreadsheet

Contains the **definitive** set of included articles after full-text reading. This is the gold standard.

**File name:** `YourProject - Listfinal.xlsx`

**Required columns:** `title`, `decision`

Only articles that survived full-text evaluation should appear here. The report measures how many of these the AI would have retained — this is the most clinically relevant metric.

### Step 4: Run AI Screening

1. Start the web app: `.venv\Scripts\python.exe -m uvicorn scripts.backend:app --reload --port 8000` (from repo root)
2. Open http://localhost:8000
3. Select a model (e.g., GPT-5.2)
4. Enter your API key
5. Write your PICO synopsis and inclusion/exclusion criteria
6. Upload your TIAB spreadsheet (the one with `title` + `abstract` columns)
7. Click **Send to Backend**
8. Wait for completion, then download the result

**Naming the output:** Rename the downloaded file to match the convention:
```
YYYYMMDD - model - 1º teste - YourProject.xlsx
```

For test-retest analysis, run the same screening again and name it `2º teste`.

### Step 5: Create Metadata Spreadsheet

Create `metadata.xlsx` in the `input/` folder with one row per AI execution:

| project | code | model | parameter | version | time_ia | time_human | tokens input | tokens_output | cost_input | cost_output | cost_total |
|---------|------|-------|-----------|---------|---------|------------|--------------|---------------|------------|-------------|------------|
| mino | 20260227 | gpt-5-mini | reasoning=medium | 1º teste | 0:04:32 | 2:00:00 | 152340 | 28450 | 0.023 | 0.114 | 0.137 |

**Column details:**

- **`project`**: Must match the project name in your file names (case-sensitive).
- **`code`**: The YYYYMMDD prefix of your AI result file.
- **`model`**: Must match the model name in your AI result file.
- **`parameter`**: Free-text description of AI parameters used.
- **`version`**: Test number — `1º teste` or `2º teste`. This identifies replication runs for the test-retest analysis.
- **`time_ia`**: How long the AI took (format: `H:MM:SS` or `HH:MM:SS`).
- **`time_human`**: Human time for TIAB screening, calculated as **1 minute per record × 2 reviewers**, expressed in researcher-hours (format: `H:MM:SS`). Used in the **Workload Reduction** analysis (Section 11).
- **`tokens input` / `tokens_output`**: Token counts from the execution, available in the **OpenAI dashboard** (Usage section).
- **`cost_input` / `cost_output` / `cost_total`**: Costs in USD (available from OpenAI usage dashboard).

### Step 6: Place All Files in `input/`

## Title Consistency Checklist

Title matching is critical for accurate results. Common pitfalls:

| Problem | Solution |
|---------|----------|
| Different capitalization | Matching is case-insensitive (handled automatically) |
| Extra whitespace | Whitespace is normalized (handled automatically) |
| Different special characters (e.g., `–` vs. `-`) | Export both human and AI data from the same source file |
| Truncated titles | Ensure full titles in both spreadsheets |
| HTML entities (e.g., `&amp;`) | Clean titles before use, or ensure consistent encoding |

**Best practice:** Use the **exact same TIAB spreadsheet** as the source for both human review and AI screening. This guarantees title consistency.

## Reusing TIAB Data

The TIAB spreadsheet serves dual purposes:

1. **Source for AI screening** — Upload it to the web app (the app uses `title` + `abstract` columns)
2. **Human reference** — The `decision` column contains the human reviewer's judgment

This means you only need **one** TIAB spreadsheet per project. The AI result file will have a `screening_decision` column added by the AI, while the original TIAB keeps the human `decision`.

## Interpreting the Report

### Key Metrics

| Metric | What It Tells You | Good Value |
|--------|-------------------|------------|
| **Sensitivity** | % of human-included articles the AI also included | ≥ 95% |
| **Specificity** | % of human-excluded articles the AI also excluded | ≥ 50% |
| **F1 Score** | Harmonic mean of Sensitivity and PPV | Higher is better |
| **Listfinal Capture** | % of final included articles the AI retained | ≥ 95% |
| **Specificity (LF)** | % of non-Listfinal articles correctly excluded | Higher is better |
| **Test-Retest Kappa** | Reproducibility of AI decisions across runs | ≥ 0.80 |
| **Workload Reduction** | Time saved using AI vs. human screening | Higher is better |
| **Efficiency Score** | Combined selectivity × capture (Section 12) | Higher is better |

## Troubleshooting

| Issue | Solution |
|-------|----------|
| `python` is not recognized | Python not added to PATH — reinstall Python and check "Add Python to PATH" on the first installer screen |
| `.venv\Scripts\pip.exe` not found | Virtual environment not created — run `python -m venv .venv` from the project root folder |
| `UnicodeEncodeError` on Windows | Run `$env:PYTHONIOENCODING="utf-8"` in the same terminal before the report command |
| `No module named 'docx'` | Run `.\.venv\Scripts\pip.exe install python-docx` |
| `No module named 'matplotlib'` | Run `.\.venv\Scripts\pip.exe install matplotlib` |
| `No module named 'fastapi'` | Run `.\.venv\Scripts\pip.exe install -r requirements.txt` |
| `Could not import module "backend"` | You must run the uvicorn command from the **project root folder** (`AI-tools/`), not from inside `scripts/` |
| Report finds no files | Check that input files follow the exact naming conventions described in [File Naming Conventions](#file-naming-conventions) |
| Metadata mismatch warnings | Ensure `project`, `code`, `model` in `metadata.xlsx` match the AI result filenames exactly (case-sensitive) |
| XLSX library not loaded (web app) | Access the app via http://localhost:8000, not by opening `index.html` directly as a file |
